# Ad Campaign Demo (Go + React + Postgres)

A small full-stack CRUD demo: create ad campaigns, log impression/click
events against them, and view aggregate stats. It is **not** a
production ad-tech platform, and the code deliberately avoids
overstating what it does.

### What this actually is
- **Backend:** Go, `net/http` + `gorilla/mux`, with connection-pool
  tuning and explicit server timeouts. Concurrency comes from Go's
  goroutine-per-request model, not from any custom scheduling.
- **Frontend:** React + Vite, styled with Tailwind's CDN build.
- **Database:** A single PostgreSQL instance with two tables
  (`campaigns`, `ad_events`). Analytics are plain `COUNT`/`SUM` SQL
  queries — there is no ClickHouse, BigQuery, or streaming pipeline
  anywhere in this project. At meaningful data volume, this table
  would need partitioning, rollups, or a dedicated analytics store.
- **Revenue:** Generated by `simulateBidRevenue()` in `handlers.go`,
  which picks a randomized value in a plausible CPM/CPC range. It does
  not model real auctions, floor prices, or advertiser demand — it's
  sample-data generation, not a bidding engine.
- **`/mcp/v1`:** A JSON-RPC 2.0 endpoint that handles `initialize`,
  `tools/list`, and `tools/call` and actually routes to real data
  queries. It is a simplified, hand-rolled implementation for
  demonstration purposes — it is not built on the official MCP SDK and
  hasn't been validated against the full MCP spec (transport
  negotiation, capability discovery details, etc.), so treat it as a
  reference sketch rather than a certified server.
- **Budget handling:** Writes to `ad_events` happen inside a
  transaction that locks the campaign row, checks remaining budget,
  and pauses the campaign if it would be exceeded — so at least the
  one piece of "real" business logic here is enforced rather than
  decorative.
- **Tests:** `backend/handlers_test.go` covers input validation, the
  success path for campaign creation, and the not-found/exhausted-budget
  paths for event tracking, using `sqlmock` so no live database is
  required to run them.

### What this is *not*
- Not a high-throughput or "web-scale" system — there's no load
  testing, no benchmarks, and no evidence behind that kind of claim.
- Not a real-time bidding or programmatic-advertising engine.
- Not a certified MCP server implementation.
- Still light on production concerns: no auth, no rate limiting, no
  structured logging/metrics, no migrations tooling (schema is created
  ad hoc on boot).

### Running locally
1. Start a local PostgreSQL instance and either set `DATABASE_URL` or
   rely on the default dev credentials (`postgres/postgres`, database
   `adtech_db`).
2. Backend:
   ```bash
   cd backend
   go mod tidy
   go run .
   ```
3. Run tests:
   ```bash
   cd backend
   go test ./...
   ```
4. Frontend:
   ```bash
   cd frontend
   npm install
   npm run dev
   ```
